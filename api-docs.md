# ParkWise API Documentation

Last updated: 2026-05-18

This document describes the current Django REST API used by the ParkWise mobile apps for user booking, society administration, guard QR validation, payments, and super-admin reporting. It is written in a Postman-friendly format: each section includes purpose, authentication, request bodies, query parameters, and representative responses.

## Base URLs

| Environment | Base URL |
| --- | --- |
| Local backend | `http://127.0.0.1:8000` |
| Android emulator | `http://10.0.2.2:8000` |
| Physical device | `http://<laptop-lan-ip>:8000` |

All application endpoints are under:

```text
{{base_url}}/api/v1
```

Built-in API artifacts:

| Method | Endpoint | Description |
| --- | --- | --- |
| GET | `/api/docs/` | Swagger UI generated from the backend schema |
| GET | `/api/schema/` | OpenAPI schema generated by drf-spectacular |

## Authentication

The API uses JWT bearer tokens from SimpleJWT.

Send authenticated requests with:

```http
Authorization: Bearer {{access_token}}
Content-Type: application/json
```

Access tokens live for 30 minutes. Refresh tokens live for 7 days and rotate on refresh.

## Common Rules

Pagination uses DRF page-number pagination:

```json
{
  "count": 25,
  "next": "http://127.0.0.1:8000/api/v1/bookings/list/?page=2",
  "previous": null,
  "results": []
}
```

Default rate limits are `30/minute` for anonymous requests and `60/minute` for authenticated users.

Common HTTP responses:

| Status | Meaning |
| --- | --- |
| `200 OK` | Request succeeded |
| `201 Created` | Resource created |
| `400 Bad Request` | Invalid payload, invalid state, or failed business rule |
| `401 Unauthorized` | Missing or invalid JWT |
| `403 Forbidden` | Authenticated user does not have the required role |
| `404 Not Found` | Resource does not exist or is not visible to the user |
| `429 Too Many Requests` | Rate limit exceeded |

## Roles and Permissions

| Role | Main Capabilities |
| --- | --- |
| `user` | Register/login, manage profile and vehicles, search societies/slots, create bookings, pay, view QR, view penalties |
| `society_admin` | Manage own society slots, approve residents/slots/guards, view society dashboard |
| `guard` | Validate entry/exit QR codes based on scan permissions |
| `super_admin` | Create/update societies and view platform-level stats |

Guard login is blocked unless the guard approval status is `approved`.

## Authentication APIs

### Register User or Society Admin

`POST {{base_url}}/api/v1/auth/register/`

Auth: public

Registers a normal user or a society admin. A user can optionally submit a `society_join_code`; this creates a pending membership request for the society admin to approve. A society admin must provide society details and gets linked to the newly created society.

Request for user:

```json
{
  "email": "user@example.com",
  "phone": "9876543210",
  "full_name": "Example User",
  "password": "Password@123",
  "role": "user",
  "society_join_code": "AB12CD34",
  "flat_number": "A-401",
  "floor_number": "4"
}
```

Request for society admin:

```json
{
  "email": "admin@example.com",
  "phone": "9876543211",
  "full_name": "Society Admin",
  "password": "Password@123",
  "role": "society_admin",
  "society_name": "Green Heights",
  "society_address": "MG Road",
  "society_city": "Mumbai",
  "society_state": "Maharashtra",
  "society_pincode": "400001",
  "society_latitude": "19.076000",
  "society_longitude": "72.877700"
}
```

Response:

```json
{
  "user": {
    "id": "uuid",
    "email": "user@example.com",
    "phone": "9876543210",
    "full_name": "Example User",
    "role": "user",
    "approval_status": "approved",
    "society": null,
    "society_name": null
  },
  "membership_status": "pending_approval",
  "tokens": {
    "refresh": "jwt-refresh-token",
    "access": "jwt-access-token"
  }
}
```

`POST {{base_url}}/api/v1/auth/login/`

Auth: public

```json
{
  "email": "user@example.com",
  "password": "Password@123"

```

Response:

```json
{
  "refresh": "jwt-refresh-token",
  "access": "jwt-access-token"

```

The JWT also contains claims for `role`, `full_name`, `approval_status`, optional `society_id`, `can_scan_entry`, and `can_scan_exit`.

`GET {{base_url}}/api/v1/auth/profile/`

```json
  "flat_number": "B-202",

```

`GET {{base_url}}/api/v1/auth/vehicles/`

`POST {{base_url}}/api/v1/auth/vehicles/`

```json

```

`GET {{base_url}}/api/v1/auth/notifications/`

`PATCH {{base_url}}/api/v1/auth/notifications/{{notification_id}}/read/`

### Society Admin Guard Credentials

`GET {{base_url}}/api/v1/auth/guards/`

`POST {{base_url}}/api/v1/auth/guards/`

`GET {{base_url}}/api/v1/auth/guards/{{guard_id}}/`

`PATCH {{base_url}}/api/v1/auth/guards/{{guard_id}}/`

Auth: `society_admin`

Create request:

```json
{
  "full_name": "Gate Guard",
  "phone": "9876543222",
  "email": "guard@example.com",
  "can_scan_entry": true,
  "can_scan_exit": true
}
```

If `email` is omitted, the API creates a local guard email. The response includes one-time temporary credentials:

```json
{
  "guard": {
    "id": "uuid",
    "email": "guard@example.com",
    "full_name": "Gate Guard",
    "can_scan_entry": true,
    "can_scan_exit": true
  },
  "credentials": {
    "email": "guard@example.com",
    "temporary_password": "generated-password"
  }
}
```

Guard permission update:

```json
{
  "can_scan_entry": true,
  "can_scan_exit": false
}
```

At least one scan permission must remain enabled.

### Search Available Societies

`POST {{base_url}}/api/v1/societies/search/`

Auth: bearer token

Searches nearby societies that have at least one available approved slot for the requested time and vehicle type.

```json
{
  "destination_text": "Bandra West",
  "destination_lat": 19.0600,
  "destination_lng": 72.8365,
  "destination_place_id": "",
  "current_lat": 19.0760,
  "current_lng": 72.8777,
  "booking_date": "2026-05-18",
  "start_time": "14:00:00",
  "duration_minutes": 120,
  "vehicle_type": "car",
  "search_radius_km": 10
}
```

Use either `end_time` or `duration_minutes`, not both. Booking duration must be at least 30 minutes and at most 24 hours. Bookings cannot be more than 24 hours in advance.

## Slot APIs

### List or Create Slots

`GET {{base_url}}/api/v1/societies/{{society_id}}/slots/`

Auth: bearer token

Optional filters:

| Query Param | Description |
| --- | --- |
| `slot_type` | `car` or `bike` |
| `state` | `available`, `reserved`, `occupied`, `blocked` |
| `owner_id` | Return slots owned by a resident |
| `booking_date` | Required for availability filter |
| `start_time` | Required for availability filter |
| `end_time` | Use with `booking_date` and `start_time` |
| `duration_minutes` | Alternative to `end_time` |
| `vehicle_type` | Required for availability filter |

Availability-filter example:

```text
GET /api/v1/societies/{{society_id}}/slots/?booking_date=2026-05-18&start_time=14:00:00&duration_minutes=120&vehicle_type=car
```

`POST {{base_url}}/api/v1/societies/{{society_id}}/slots/`

Auth: authenticated user in the same society

Society admins create approved society-owned slots. Residents create resident-owned slots in `pending` approval status and `blocked` state.

```json
{
  "slot_number": "A-101",
  "floor": "1",
  "slot_type": "car",
  "hourly_rate": "50.00",
  "available_from_write": "08:00:00",
  "available_to_write": "22:00:00"
}
```

## Booking APIs

### Create Booking

`POST {{base_url}}/api/v1/bookings/`

Auth: `user`

Creates a `pending_payment` booking lock for 5 minutes. The slot is not reserved until payment is captured.

```json
{
  "slot_id": "{{slot_id}}",
  "vehicle_id": "{{vehicle_id}}",
  "start_time": "2026-05-18T14:00:00+05:30",
  "end_time": "2026-05-18T16:00:00+05:30"
}
```

Response includes `booking_number`, `amount`, `status`, `payment_status`, and `lock_expires_at`.

### List Bookings

`GET {{base_url}}/api/v1/bookings/list/`

Auth: bearer token

Optional query:

| Query Param | Description |
| --- | --- |
| `status` | Filter by booking status |

Users see their own bookings. Society admins and guards see bookings in their society.

### Booking Detail

`GET {{base_url}}/api/v1/bookings/{{booking_id}}/`

Auth: bearer token

When a booking is still `pending_payment`, this endpoint attempts to auto-sync pending Stripe or Razorpay payments before returning the booking. This helps the UI reflect completed payments after the user returns from checkout or refreshes the booking screen.

### Cancel Booking

`POST {{base_url}}/api/v1/bookings/{{booking_id}}/cancel/`

Auth: booking owner

Only `pending_payment` and `confirmed` bookings can be cancelled. Cancelling a confirmed booking frees a reserved slot.

### Booking QR Code

`GET {{base_url}}/api/v1/bookings/{{booking_id}}/qr/`

Auth: booking owner

Returns `image/png`. QR is available only for `confirmed` or `active` bookings.

## Payment APIs

### Initiate Booking Payment

`POST {{base_url}}/api/v1/payments/initiate/`

Auth: booking owner

Creates a payment record for a `pending_payment` booking.

Stripe hosted checkout:

```json
{
  "booking_id": "{{booking_id}}",
  "gateway": "stripe",
  "embedded": false
}
```

Stripe embedded checkout:

```json
{
  "booking_id": "{{booking_id}}",
  "gateway": "stripe",
  "embedded": true
}
```

Razorpay:

```json
{
  "booking_id": "{{booking_id}}",
  "gateway": "razorpay"
}
```

Stripe response fields:

```json
{
  "id": "payment-uuid",
  "booking": "booking-uuid",
  "payment_type": "booking",
  "amount": "100.00",
  "currency": "INR",
  "provider": "stripe",
  "stripe_checkout_session_id": "cs_test_xxx",
  "status": "created",
  "checkout_url": "https://checkout.stripe.com/...",
  "checkout_client_secret": "cs_test_xxx_secret_xxx",
  "stripe_publishable_key": "<publishable-key-for-embedded-checkout>"
}
```

For embedded checkout, the app should use `checkout_client_secret` with the Stripe embedded checkout UI. This avoids opening a new tab. When the payment completes, the Stripe webhook and/or verification endpoint changes the payment from `created` to `captured`, the booking from `pending_payment` to `confirmed`, and the slot from `available` to `reserved`.

### Verify Stripe Payment

`POST {{base_url}}/api/v1/payments/verify/`

Auth: booking owner

```json
{
  "checkout_session_id": "{{checkout_session_id}}"
}
```

If Stripe reports the session as paid, the API returns the captured payment and updates related booking/slot state.

### Verify Razorpay Payment

`POST {{base_url}}/api/v1/payments/razorpay/verify/`

Auth: booking owner

```json
{
  "razorpay_order_id": "order_xxx",
  "razorpay_payment_id": "pay_xxx",
  "razorpay_signature": "signature"
}
```

### Stripe Webhook

`POST {{base_url}}/api/v1/payments/webhook/`

Auth: Stripe webhook signature in `Stripe-Signature`

Handled events:

| Event | Behavior |
| --- | --- |
| `checkout.session.completed` | Verifies checkout, marks payment `captured`, confirms booking or pays penalty |
| `payment_intent.payment_failed` | Marks payment `failed` when a matching payment exists |

In development, if `STRIPE_WEBHOOK_SECRET` is empty, the backend parses the event without signature verification. Production must set `STRIPE_WEBHOOK_SECRET`.

### Manual Stripe Verification

`POST {{base_url}}/api/v1/payments/manual-verify/`

Auth: booking owner

Testing helper for manually verifying a Stripe checkout session:

```json
{
  "checkout_session_id": "{{checkout_session_id}}"
}
```

## QR Validation APIs

### Entry Scan

`POST {{base_url}}/api/v1/qr/entry/`

Auth: `guard` with `can_scan_entry=true`

```json
{
  "qr_token": "signed-booking-qr-token"
}
```

On success, the booking changes from `confirmed` to `active`, and the slot changes to `occupied`.

Success response:

```json
{
  "status": "entry_granted",
  "booking_id": "uuid",
  "booking_number": "BK-20260518-ABC123",
  "slot_number": "A-101",
  "vehicle_number": "MH01AB1234",
  "owner_name": "Example User",
  "payment_status": "captured",
  "entry_time": "2026-05-18T14:05:00+05:30",
  "exit_time": null,
  "overstay": false,
  "penalty": null
}
```

### Exit Scan

`POST {{base_url}}/api/v1/qr/exit/`

Auth: `guard` with `can_scan_exit=true`

```json
{
  "qr_token": "signed-booking-qr-token"
}
```

On success, the booking changes from `active` to `completed`, and the slot changes to `available`. If the exit happens after `end_time`, an overstay penalty is created and returned.

## Main Workflows

### User Booking and Payment Flow

1. `POST /api/v1/auth/register/` or `POST /api/v1/auth/login/`.
2. `POST /api/v1/auth/vehicles/` to add a vehicle.
3. `POST /api/v1/societies/search/` to find societies with available slots.
4. `GET /api/v1/societies/{{society_id}}/slots/` with availability filters.
5. `POST /api/v1/bookings/` to create a `pending_payment` booking.
6. `POST /api/v1/payments/initiate/` to create a Stripe or Razorpay payment.
7. Complete payment in the app.
8. Payment completion is reflected through webhook and/or `POST /api/v1/payments/verify/`.
9. `GET /api/v1/bookings/{{booking_id}}/` returns `status=confirmed` and `payment_status=captured`.
10. `GET /api/v1/bookings/{{booking_id}}/qr/` returns the entry/exit QR code.

### Guard Entry and Exit Flow

1. Guard logs in after society admin approval.
2. Guard scans the booking QR token.
3. `POST /api/v1/qr/entry/` validates entry and changes the booking to `active`.
4. `POST /api/v1/qr/exit/` validates exit and changes the booking to `completed`.
5. The society admin dashboard reflects active/completed bookings and recent gate activity.

### Society Admin Approval Flow

1. User registers with a `society_join_code`.
2. Society admin calls `GET /api/v1/societies/{{society_id}}/join-requests/?status=pending`.
3. Society admin calls the join request decision endpoint.
4. Resident can submit a slot through `POST /api/v1/societies/{{society_id}}/slots/`.
5. Society admin approves or rejects the resident slot using the slot decision endpoint.
